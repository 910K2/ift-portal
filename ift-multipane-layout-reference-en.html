01 Resizable panes: should the designer decide?

Answer: Yes — the designer decides. But the decision must be justified.

Approach Appropriate context Rationale
Fixed layout General-user FMS, corporate homepage Reduces cognitive load. Keeps users focused.
Resizable panes Power-user monitoring, map + table side-by-side Efficiency-first. Pro-tool and IDE paradigm.
Preset switching FMS v16 approach Compromise between flexibility and UX stability.

In the IFT FMS context, the ratio of vehicle list width to map width is something users genuinely want to control (e.g. field operations on narrow screens). However, "anything goes" causes confusion. 2–3 preset layout modes with a toggle is the pragmatic solution.

02 Responsive grid: is it needed on a homepage too?

Yes — and the homepage is arguably where it matters most.

Core technical concept

4K (2560px+) → 6-column grid Desktop (1280px) → 4-column grid Tablet (768px) → 2-column grid Mobile (360px) → 1-column grid

Achievable in CSS alone:

CSS
.service-grid { display: grid; grid-template-columns: repeat(auto-fit, minmax(200px, 1fr)); gap: 1.5rem; }

The auto-fit + minmax combination achieves automatic optimization without explicitly writing breakpoints. This is part of the direct answer to Gopi's "1024px limitation."

03 Gopi's "1024px limitation"

This is incorrect — or reflects outdated knowledge.

Claim Accurate assessment
"Cannot support beyond 1024px" ✕ Wrong  CSS Grid has no such limit
"Mobile and PC require separate sites" ✕ Outdated  Early 2010s thinking
"4K requires special handling" △ Partially true  Addressed with max-width and clamp()

Modern approach:

CSS
/* One stylesheet handles 4K down to 320px */ .container { width: min(1400px, 100% - 2rem); margin-inline: auto; }

If Gopi's knowledge stands at this level, concern about entrusting him with the IFT website modernization is fully justified.

04 AI-adaptive layout: technically feasible?

Yes — but there are distinct levels.

Level Technology Feasibility for IFT
L1 CSS auto-fit / Container Queries Deployable now. Static rules, fully automated.
L2 JS adaptation from user behavior (surface frequent features) Feasible. LocalStorage + simple logic.
L3 AI layout optimization via user profile Technically possible. Over-investment for FMS scale.
L4 Real-time AI layout generation Research stage. Production cost too high.

IFT's pragmatic target: L1–L2. L3 and above do not justify ROI at current scale.

05 Web vs. mobile: differences in drag interaction
Interaction Web (PC) Mobile
Drag to resize pane ✓ Natural, reliable ✕ Conflicts with touch gestures
Drag to reorder grid items ✓ Drag & Drop API △ Possible via touch-action, but complex
Swipe to switch pane ✕ Unnatural on desktop ✓ Native mobile UX

Differentiating interaction design between PC and mobile is the correct approach. Delivering both from a single React codebase is the modern best practice — and should be reflected in IFT's design specifications.

06 FMS v12 → v14 → v16 — version context
Version Technology Layout flexibility Key concern
v12 (.NET) Server-side rendering Fixed. Difficult to change. Legacy dependency
v14 (React) Component-based Flexible — design-dependent Depends on design quality
v16 (v14 + feature selection) React + state management Feature selection ○, layout △ This is the core of the discussion

The absence of layout flexibility in v16 is a design decision, not a technical constraint. React is fully capable of implementing it. It is a question of priority.
